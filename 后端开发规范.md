# 星链项目后端开发规范（V2）

适用范围：`xinglian-server`（Node.js + Express + TypeScript + MySQL）  
目标：保证后端在大项目迭代中保持**可扩展、可维护、可协作**。

---

## 1. 总体原则

1. 业务优先，架构先行：新增功能必须遵循分层结构，不允许把业务堆在路由文件。
2. 向后兼容优先：变更接口时先兼容旧参数，再计划下线。
3. 单一职责：每一层只做自己的事情，避免跨层耦合。
4. 失败可观测：所有异常必须可定位（状态码、错误信息、日志上下文）。
5. 禁止隐式约定：角色、状态、错误码等必须显式定义。

---

## 2. 目录与分层规范

推荐目录（按模块组织）：

```text
src/
  app.ts
  index.ts
  config/
  core/
    errors/
  middlewares/
  modules/
    auth/
      auth.routes.ts
      auth.controller.ts
      auth.service.ts
      auth.repository.ts
      auth.types.ts
```

分层职责：

- `routes`：声明路由 + 绑定中间件，不写业务逻辑。
- `controller`：参数解析、基础校验、调用 service、统一返回结构。
- `service`：核心业务编排、规则判断、跨 repository 协作。
- `repository`：仅负责数据库访问与事务，不放业务判断。
- `types`：模块内部 DTO/响应类型/枚举映射。

---

## 3. 接口设计规范

1. 路径统一以 `/api` 开头。
2. 鉴权接口统一使用 `Authorization: Bearer <token>`。
3. 返回结构必须统一：
   - 成功：`{ ok: true, ...data }`
   - 失败：`{ ok: false, code, message, traceId }`
4. 对外字段使用稳定命名，避免频繁改名。
5. 新接口必须同步更新 `接口文档.txt`（至少：路径、方法、入参、返回示例）。

---

## 4. 参数与校验规范

1. 入参必须通过 DTO + `validate` 中间件校验（默认使用 zod）。
2. 枚举值（如 `role`）必须做白名单校验，不信任前端。
3. 禁止把未校验参数直接传入 SQL。
4. 兼容参数策略：
   - 新字段优先（如 `role`）
   - 旧字段兼容（如 `identity`）
   - 文档标注过渡期，后续移除。

---

## 5. 错误处理规范

1. 业务错误使用 `AppError(message, statusCode, code)` 抛出。
2. 全局 `errorHandler` 统一格式化错误响应。
3. 常用状态码：
   - `400` 参数错误/业务校验失败
   - `401` 未登录或 token 无效
   - `404` 资源不存在
   - `409` 资源冲突（如手机号已被占用）
   - `500` 服务器内部错误
   - `502` 第三方服务失败（如微信接口）

---

## 6. 数据库规范

1. SQL 统一放 repository，禁止 controller/service 直接写 SQL。
2. 涉及多表一致性必须使用事务。
3. 字段命名统一 snake_case，业务枚举统一 tinyint。
4. 结构变更必须有迁移脚本（后续建议接入 migration 工具）。
5. 删除策略：
   - 用户主数据谨慎硬删；
   - 涉及审计场景优先软删（保留 `deleted_at`）。

---

## 7. 鉴权与安全规范

1. JWT 必须包含最小必要信息（`userId/openid/role`）。
2. `JWT_SECRET` 仅存在服务端环境变量，严禁写死到代码仓库。
3. 所有需要登录的接口必须挂 `requireAuth`。
4. 不允许仅依赖前端传入的 `userId` 做权限判断，必须以后端鉴权信息为准。

---

## 8. 配置与环境规范

1. 所有配置从 `env.ts` 统一读取。
2. `.env.example` 必须与代码保持同步。
3. 新增环境变量必须：
   - 在 `env.ts` 增加校验；
   - 在 `.env.example` 增加示例；
   - 在文档标注用途。

---

## 9. 日志与可观测性（当前阶段要求）

1. 关键流程打印结构化日志（登录、注册完成、绑定手机号、注销）。
2. 第三方失败日志需包含 `errcode/errmsg`。
3. 日志中禁止打印敏感信息（完整身份证号、完整 token、完整手机号）。
4. 每个请求都应具备 `traceId`，并在错误响应中返回。

---

## 10. 代码风格与提交要求

1. TypeScript `strict` 模式下开发，不用 `any` 逃避类型。
2. 函数命名语义化：`findXxx`、`createXxx`、`updateXxx`、`deleteXxx`。
3. 单函数不宜过长（建议 < 80 行，超出需拆分）。
4. 每次改动要保证：
   - `npm run build` 通过；
   - 无新增 lints 错误；
   - 接口文档已同步。

---

## 11. 新功能开发流程（必须遵守）

1. 明确模块归属（auth/user/order/...）。
2. 先补类型与接口文档草稿。
3. 按 `repository -> service -> controller -> routes` 顺序实现。
4. 接入鉴权和参数校验。
5. 自测成功后再联调前端。
6. 最后更新 `接口文档.txt` 与必要说明。

---

## 12. 给 AI 助手的固定指令（建议每次新窗口先发）

可直接复制下面这段：

```text
你现在是本项目后端协作助手，请严格遵守《后端开发规范.md》。
要求：
1) 按 modules 分层实现：routes/controller/service/repository/types。
2) 不要把业务逻辑写在 routes。
3) 所有需要登录的接口必须使用 JWT 鉴权。
4) 入参严格校验，枚举字段做白名单校验。
5) 业务错误使用 AppError，统一错误响应。
6) 变更接口时同步更新 接口文档.txt。
7) 完成后执行 npm run build，并确保无新增 lints 问题。
```

---

## 13. 控制器约束（V2新增）

1. controller 禁止手写分散参数校验，必须依赖 DTO 校验结果。
2. controller 不直接拼装失败 JSON，统一使用 `fail()` 或抛 `AppError`。
3. controller 成功返回统一使用 `success()`。

---

## 14. 兼容策略（V2新增）

1. 兼容参数（如 `identity`）必须在接口文档标注“过渡字段”。
2. 每个兼容字段要标注计划移除时间，避免长期债务。
